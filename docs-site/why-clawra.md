# 为什么选 Clawra？

## 问题：Prompt Engineering 的尽头是地狱

用 LangChain 6 个月后，你的 Prompt 变成了 2000 行。偶尔能跑，但说不清哪个规则和哪个冲突，一调就崩。

这不是工程能力问题。**这是架构问题。**

```
规则 = 结构化的、确定性的知识
Prompt = 非结构化的、概率性的表达
把前者塞进后者 → 迟早出事
```

## 解决思路：神经符号融合

Clawra 的核心设计假设是：**知识应该用结构化方式存储，用形式化方式执行。**

```
文本输入 → LLM 提取 → 知识图谱 → 符号逻辑推理
                                    ↓
                             硬性约束拦截
```

不是 Prompt 提醒，是**数学证明级别的拦截**。

## 实际案例

=== "LangChain 方式"
    ```python
    system_prompt = """
    当讨论燃气压力时：
    - 禁止推荐超过 0.4MPa 的压力值
    - 超过 0.35MPa 必须警告爆炸风险
    ...
    """
    # 6个月后变成2000行，没人敢动
    ```

=== "Clawra 方式"
    ```python
    clawra.learn("燃气调压箱出口压力不得超过 0.4MPa")

    # Clawra 自动：
    # 1. 提取: pressure ≤ 0.4MPa
    # 2. 注册为硬性约束
    # 3. 生成反向验证链
    # 4. LLM建议 pressure=0.8MPa → 直接拦截
    ```

## 8 阶段进化闭环

```
感知 → 学习 → 推理 → 执行 → 评估 → 漂移检测 → 规则修订 → 知识更新
  ↑__________________________________________________________________|
                          (从错误中学习)
```

每犯一次错，Clawra 更新自己的规则集。**这不是一次性的流水线，是活着的系统。**

## 适合谁

✅ 规则高频变更（合规、监管、政策）
✅ 领域专业知识要求高（医疗、法律、工业）
✅ 安全要求严格（关键决策零幻觉）
✅ 需要 AI 持续进化

❌ 快速原型（用 LangChain）
❌ 简单问答场景
