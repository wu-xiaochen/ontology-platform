### 2.2 推理引擎服务 - 详细设计

#### RESTful API 端点规划

```
/api/v1/
├── /ontologies          # 本体管理
│   ├── POST /           # 创建本体
│   ├── GET /            # 获取本体列表
│   ├── POST /import     # 导入本体
│   └── GET /{id}/export # 导出本体
├── /entities            # 实体管理
│   ├── POST /           # 创建实体
│   ├── GET /            # 获取实体列表
│   └── PUT /{id}/confidence # 更新置信度
├── /relations           # 关系管理
├── /reason              # 推理服务
│   ├── POST /           # 执行推理
│   ├── GET /{id}/chain  # 获取推理链
│   └── POST /confirm    # 确认假设
├── /graph               # 知识图谱
│   ├── GET /            # 图谱查询
│   └── POST /path       # 路径查询
├── /extract             # 抽取服务 (v3.3)
│   ├── POST /           # 抽取实体
│   ├── POST /preview    # 预览抽取
│   └── POST /confirm    # 确认抽取
└── /analytics           # 数据分析
```

#### GraphQL 查询接口

| 查询类型 | 描述 | 示例 |
|----------|------|------|
| ontology | 本体查询 | `query { ontology(id: "xxx") { name entities { name } } }` |
| entities | 实体查询 | `query { entities(type: "设备") { name confidence } }` |
| relations | 关系查询 | `query { relations(type: "包含") { source { name } target { name } } }` |
| reasoning | 推理查询 | `query { reason(input: "报警器故障") { finalConclusion confidence } }` |
| graph | 图谱查询 | `query { graph(entityTypes: ["设备"]) { nodes { name } edges { type } } }` |

### 2.3 知识图谱模块 - 详细设计

#### Neo4j 数据模型

```
(实体:Entity {
  id: UUID,
  name: String,
  type: String,
  properties: Map,
  confidence: String,
  ontology_id: UUID
})-[:关系:RELATION {
  id: UUID,
  type: String,
  properties: Map,
  confidence: Float
}]->(实体:Entity)
```

#### 图查询能力

| 功能 | 描述 | Cypher示例 |
|------|------|------------|
| 节点搜索 | 按名称/类型搜索实体 | `MATCH (e:Entity {type: '设备'}) RETURN e` |
| 关系探索 | 点击节点显示关联关系 | `MATCH (e {name: '报警器'})-[r]->(related) RETURN related` |
| 路径查询 | 查询两实体间最短路径 | `MATCH p = shortestPath((a)-[*]->(b)) RETURN p` |
| 子图查询 | 按条件导出子图 | `MATCH p = (root)-[*1..3]->() WHERE root.name = '燃气系统' RETURN p` |
| 统计分析 | 图谱统计信息 | `MATCH (e) RETURN count(e), labels(e)` |

### 2.4 用户交互 - 详细设计

#### 置信度标注展示

| 置信度 | 颜色 | 含义 | 用户行动 |
|--------|------|------|----------|
| CONFIRMED | 🟢 绿色 | 已验证可靠的结论 | 可直接采纳 |
| ASSUMED | 🟡 黄色 | 基于合理假设的结论 | 需关注假设条件 |
| SPECULATIVE | 🟠 橙色 | 推测性结论 | 谨慎采纳 |
| UNKNOWN | ⚪ 灰色 | 无法确定的结论 | 需补充信息 |

#### 假设声明与确认流程

```
┌─────────────────────────────────────────────────────────────┐
│                    假设确认流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   推理引擎 ──────→ 检测假设 ──────→ 暂停推理                │
│       │                │              │                    │
│       │                ▼              ▼                    │
│       │         展示假设内容    发送确认请求到UI             │
│       │                │              │                    │
│       │                ▼              ▼                    │
│       │         需要用户确认    等待用户确认                │
│       │                │              │                    │
│       ▼                ▼              ▼                    │
│   ┌─────────┐    ┌───────────┐   ┌─────────────┐           │
│   │ 继续推理 │◀───│ 用户确认  │───│ 用户拒绝    │           │
│   └─────────┘    └───────────┘   └─────────────┘           │
│       │                                                    │
│       ▼                                                    │
```

### 2.5 v3.3 核心特性 - 安全边界控制

#### 自动学习安全机制

| 功能 | 默认状态 | 安全说明 |
|------|----------|----------|
| 自动抽取到本体 | 🔴 **禁用** | 用户说"写入本体"才触发，需确认 |
| 自动置信度升级 | 🔴 **禁用** | 用户说"确认正确"才触发，需确认 |
| 读取工作区文件 | 🟡 **受限** | 仅读取 memory/ 目录 |
| 写入本地本体 | 🔴 **禁用** | 用户确认后单次执行 |
| 第三方API调用 | 🔴 **禁用** | 需用户明确授权 |

#### 安全控制流程

```yaml
# 写入本体安全流程
WRITE_ONTOLOGY_FLOW:
  触发条件: 用户说"写入本体"或"记录这个实体"
  步骤1: 系统识别可抽取内容
  步骤2: 展示给用户预览确认 ⚠️ 关键步骤
  步骤3: 用户确认"是的，记录它"
  步骤4: 写入本体
  步骤5: 反馈用户"已写入本体：xxx"

# 置信度升级安全流程
CONFIDENCE_UPGRADE_FLOW:
  触发条件: 用户说"确认这个是对的"或"这个答案正确"
  步骤1: 系统生成升级建议
  步骤2: 展示给用户确认 ⚠️ 关键步骤
  步骤3: 用户确认
  步骤4: 升级置信度 (ASSUMED → CONFIRMED)
  步骤5: 反馈用户
```

## ⚠️ 风险与假设

### 主要风险

1. **技术风险**: OWL/RDF解析复杂度可能超出预期
2. **集成风险**: Neo4j大规模数据性能调优
3. **产品风险**: 用户对可解释性推理的接受度
4. **安全风险**: v3.3自动学习功能用户学习成本

### 关键假设

1. ontology-clawra v3.3 功能稳定
2. 团队具备图数据库和本体论经验
3. 市场需求保持稳定增长
