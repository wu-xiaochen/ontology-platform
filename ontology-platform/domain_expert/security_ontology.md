# 网络安全领域本体设计

## 概述

本文档定义网络安全领域的核心本体模型，用于指导信息安全知识库构建、威胁分析和智能化安全运营。

---

## 一、概念定义 (Concepts)

### 1.1 核心实体

| 概念 | 英文 | 定义 | 关键属性 |
|------|------|------|----------|
| 资产 | Asset | 组织拥有的具有价值的信息资源，包括硬件、软件、数据、服务等 | 资产编号、名称、类型、价值、位置、负责人、敏感等级 |
| 威胁 | Threat | 可能对资产造成损害的潜在事件或攻击行为 | 威胁编号、名称、类型、来源、目标、概率、影响 |
| 漏洞 | Vulnerability | 系统或应用中存在的安全弱点，可被威胁利用 | 漏洞编号、名称、CVE编号、严重程度、影响范围、修复状态 |
| 风险 | Risk | 威胁利用漏洞对资产造成影响的可能性和后果 | 风险编号、资产、威胁、漏洞、风险值、等级、处理状态 |
| 攻击向量 | Attack Vector | 攻击者用来访问系统或网络的路径或方法 | 向量编号、名称、类型、复杂度、关联漏洞 |
| 安全事件 | Security Event | 系统中发生的可能影响安全性的事件 | 事件编号、时间、类型、来源、目标、状态、严重程度 |
| 安全事件响应 | Incident Response | 对安全事件进行检测、分析、遏制、根除、恢复的过程 | 响应编号、事件、阶段、处置人、处置措施、耗时 |
| 防护措施 | Security Control | 用于保护资产免受威胁的技术或管理手段 | 措施编号、名称、类型、实施位置、有效性、状态 |
| 渗透测试 | Penetration Test | 模拟攻击者行为的安全测试，评估系统安全性 | 测试编号、范围、方法、发现数、风险等级、报告日期 |
| 安全策略 | Security Policy | 组织关于信息安全的规则、标准和指南 | 策略编号、名称、适用范围、状态、审批人、生效日期 |
| 合规要求 | Compliance Requirement | 法律法规、行业标准或组织内部规定的安全要求 | 要求编号、名称、来源、适用范围、合规状态、审计周期 |
| 身份认证 | Authentication | 验证用户或系统身份真实性的过程 | 认证编号、方式、用户、时间、地点、结果 |
| 访问控制 | Access Control | 限制对资源的访问权限的管理机制 | 控制编号、主体、客体、权限、策略、时间范围 |
| 加密 | Encryption | 使用算法保护数据机密性的技术 | 加密编号、算法、密钥长度、应用场景、保护级别 |
| 安全审计 | Security Audit | 对系统安全状况进行检查和评估的活动 | 审计编号、范围、方法、发现项、风险等级、报告日期 |

## 二、规律 (Loirs)

### 2.1 ATT&CK框架

**定义**: MITRE ATT&CK是一个全球可访问的威胁行为知识库，用于描述和分类恶意行为。

**核心矩阵**:

| 战术阶段 | 描述 | 常见技术 |
|----------|------|----------|
| 侦察 | 收集信息以规划未来攻击 | 主动扫描、收集信息 |
| 资源开发 | 建立攻击所需资源 | 获取基础设施、获取能力 |
| 初始访问 | 获得网络入口点 | 钓鱼、漏洞利用、合法凭证 |
| 执行 | 运行恶意代码 | 脚本执行、命令执行 |
| 持久化 | 保持长期访问 | 启动项、账户、计划任务 |
| 权限提升 | 获取更高权限 | 滥用提权、逃逸 |
| 防御规避 | 规避检测 | 混淆、伪装、伪装 |
| 凭证访问 | 窃取凭证 | 键盘记录、凭证转储 |
| 内部侦察 | 了解环境 | 收集文件、查询数据库 |
| 横向移动 | 在环境中移动 | 远程服务、Pass-the-Hash |
| 数据收集 | 收集目标数据 | 收集快照、压缩数据 |
| 命令与控制 | 与被控系统通信 | 协议隧道、隐蔽通道 |
| 数据泄露 | 将数据带出 | 自动泄露、协议隧道 |
| 影响 | 中断可用性 | 数据销毁、服务停止 |

### 2.2 Kill Chain模型

**定义**: 网络攻击生命周期模型，描述攻击从侦察到目标的各个阶段。

**攻击链阶段**:

```
1. 侦察 (Reconnaissance)
   └── 目标选择、信息收集

2. 武器化 (Weaponization)
   └── 构建恶意软件、创建漏洞利用

3. 投递 (Delivery)
   └── 钓鱼邮件、恶意链接、移动介质

4. 利用 (Exploitation)
   └── 触发漏洞、获取初始访问

5. 安装 (Installation)
   └── 安装恶意软件、建立持久化

6. 命令与控制 (Command & Control)
   └── 建立隐蔽通信通道

7. 目标达成 (Actions on Objectives)
   └── 数据窃取、破坏活动、持续监控
```

**防御策略**: 在攻击链的每个阶段部署相应的防御措施，越早拦截效果越好。

### 2.3 纵深防御原则

**定义**: 在多个层面部署多种安全控制，形成多层防护体系。

**防护层次**:

| 层次 | 防护内容 | 典型措施 |
|------|----------|----------|
| 物理安全 | 物理访问控制 | 门禁、监控、机房安全 |
| 网络安全 | 网络边界防护 | 防火墙、入侵检测、网络分段 |
| 主机安全 | 终端和服务器安全 | 杀毒、补丁管理、HIDS |
| 应用安全 | 软件层面安全 | 安全开发、代码审计、WAF |
| 数据安全 | 数据保护 | 加密、脱敏、访问控制 |
| 身份安全 | 身份认证授权 | MFA、IAM、零信任 |

### 2.4 最小权限原则

**定义**: 用户和程序只应获得完成任务所需的最小权限，减少攻击面。

**应用场景**:

- 账户权限：按角色分配权限，避免过度授权
- 文件权限：遵循最小读取/写入原则
- 网络权限：仅开放必要端口，限制访问范围
- API权限：按需授权，支持权限撤销

### 2.5 零信任原则

**定义**: 不信任任何用户或设备，始终验证，持续监控。

**核心要素**:

- 永不信任，始终验证
- 最小权限访问
- 微分段隔离
- 持续监控和评估
- 全面保护数据

## 三、业务规则 (Rules)

### 3.1 漏洞管理规则

**漏洞生命周期**:

| 阶段 | 描述 | 负责人 | 时限 |
|------|------|--------|------|
| 发现 | 安全扫描、渗透测试、情报发现漏洞 | 安全团队 | - |
| 验证 | 确认漏洞真实存在，评估影响范围 | 安全团队 | 24小时 |
| 评级 | 根据CVSS和业务影响确定严重等级 | 安全团队 | 24小时 |
| 修复 | 开发/运维实施漏洞修复 | 开发/运维 | 按等级 |
| 验证 | 确认漏洞已修复 | 安全团队 | 修复后24小时 |
| 关闭 | 记录归档，更新漏洞库 | 安全团队 | - |

**修复时限**:

| 严重等级 | 响应时间 | 修复时限 | 临时缓解 |
|----------|----------|----------|----------|
| Critical | 24小时 | 7天 | 48小时 |
| High | 72小时 | 14天 | 7天 |
| Medium | 1周 | 30天 | 14天 |
| Low | 1月 | 90天 | 30天 |

### 3.2 事件响应规则

**事件分级**:

| 级别 | 描述 | 示例 | 响应团队 | 响应时间 |
|------|------|------|----------|----------|
| P0-紧急 | 业务中断、数据大规模泄露 | 勒索软件、核心系统被攻破 | 安全组+管理层 | 15分钟 |
| P1-高 | 重要系统受影响、敏感数据访问 | 账户被入侵、重要数据访问异常 | 安全组 | 1小时 |
| P2-中 | 部分系统受影响 | 钓鱼攻击、轻微漏洞利用 | 安全组 | 4小时 |
| P3-低 | 尝试性攻击、安全告警 | 扫描行为、暴力破解尝试 | 运维 | 24小时 |

**响应流程**:

```
事件检测 → 事件分类 → 事件定级 → 遏制 → 根除 → 恢复 → 事后分析 → 改进
```

**关键指标**:

| 指标 | 目标 |
|------|------|
| MTTD (平均检测时间) | <24小时 |
| MTTR (平均响应时间) | <4小时 |
| MTTR (平均恢复时间) | <8小时 |

### 3.3 访问控制规则

**身份认证方式分级**:

| 安全级别 | 认证方式 | 适用场景 | 典型应用 |
|----------|----------|----------|----------|
| L1 | 单因素认证 | 低风险 | 用户名+密码 |
| L2 | 双因素认证 | 一般安全 | 用户名+密码+短信验证码 |
| L3 | 强多因素认证 | 高安全 | 用户名+密码+硬件令牌+生物识别 |
| L4 | 持续认证 | 极高安全 | 行为分析+持续验证 |

**权限管理原则**:

- 最小权限原则
- 职责分离原则
- 权限审查周期：90天
- 超级管理员人数 ≤ 3人
- 敏感操作需要审批

### 3.4 数据安全规则

**数据分类**:

| 分类 | 定义 | 示例 | 保护要求 |
|------|------|------|----------|
| 公开 | 可公开访问的信息 | 官网信息、产品资料 | 完整性保护 |
| 内部 | 仅内部人员访问 | 内部流程、通讯录 | 访问控制 |
| 机密 | 敏感业务数据 | 源代码、合同、财务 | 加密+审计 |
| 绝密 | 高度敏感数据 | 用户密码、密钥、个人隐私 | 最高级别保护 |

**数据生命周期**:

| 阶段 | 保护措施 |
|------|----------|
| 创建 | 分类分级、标记 |
| 存储 | 加密、访问控制 |
| 使用 | 脱敏、水印 |
| 传输 | 加密、通道保护 |
| 归档 | 完整性验证、安全存储 |
| 销毁 | 安全擦除、记录 |

### 3.5 渗透测试规则

**测试类型**:

| 类型 | 频率 | 范围 | 执行团队 |
|------|------|------|----------|
| 外部渗透测试 | 每年1次 | 互联网暴露面 | 外部专业团队 |
| 内部渗透测试 | 每年1次 | 内部网络 | 内部团队/外部团队 |
| Web应用测试 | 每次发布前 | 新上线应用 | 安全团队 |
| 红蓝对抗 | 每半年1次 | 全范围 | 内部红蓝队 |
| 代码审计 | 持续 | 关键代码 | 安全团队/开发团队 |

**测试范围限制**:

- 不允许进行物理攻击测试
- 不允许进行社会工程攻击（除专项演练）
- DoS测试需要提前报批
- 关键系统需要制定回滚方案

### 3.6 合规检查规则

**主要合规框架**:

| 框架 | 适用领域 | 审计周期 | 关键要求 |
|------|----------|----------|----------|
| ISO 27001 | 信息安全管理 | 每年 | 体系认证、风险评估 |
| GDPR | 个人信息(欧盟) | 持续 | 数据保护、隐私合规 |
| 等保2.0 | 中国关键信息基础设施 | 每2年 | 等级保护、测评 |
| SOC 2 | 云服务/IT服务 | 每年 | 安全、可用、保密 |
| PCI DSS | 支付卡数据 | 每年 | 支付安全 |

**合规检查清单**:

| 检查项 | 检查内容 | 检查方式 | 频率 |
|--------|----------|----------|------|
| 账户审计 | 异常账户、特权账户 | 自动+手动 | 每月 |
| 访问审计 | 权限变更、敏感访问 | 日志审计 | 每周 |

## 七、应用场景

1. **威胁情报分析**: 基于本体知识进行威胁关联分析、攻击链还原
2. **漏洞管理**: 建立漏洞发现、评估、修复、验证的闭环流程
3. **风险评估**: 基于资产、威胁、漏洞的关联进行量化风险评估
4. **事件响应**: 根据事件类型自动匹配响应流程，提供处置指引
5. **合规管理**: 对照法规标准自动检查安全合规性
6. **安全态势感知**: 基于本体模型进行安全态势分析和预测
7. **安全运营**: 自动化安全运营、SOAR剧本编排
